Docker

La brique d'orchestration des clusters distribués : conteneurs, images, volumes, réseaux, et Docker Compose — la technologie utilisée pour simuler un cluster Citus dans le TP.

Définition Docker

Plateforme de conteneurisation permettant de packager une application et ses dépendances dans un conteneur léger, portable et reproductible. Chaque conteneur a son propre système de fichiers, ses propres processus et son propre réseau — sans VM complète.

Concepts fondamentaux

Image

Modèle en lecture seule décrivant un système de fichiers et ses métadonnées. Construite depuis un Dockerfile. Immuable — chaque modification crée un nouveau layer.

Ex : postgres:15, citusdata/citus, python:3.11

Conteneur

Instance en cours d'exécution d'une image. Léger, isolé, éphémère par défaut. Plusieurs conteneurs peuvent tourner depuis la même image.

Volume

Mécanisme de persistance des données en dehors du cycle de vie du conteneur. Un conteneur supprimé sans volume perd toutes ses données.

Named volume Bind mount

Réseau (Network)

Les conteneurs sur le même réseau Docker se voient par leur nom de service (DNS interne). Isolés des conteneurs hors du réseau.

bridge (défaut) host overlay (Swarm)

Registry

Dépôt d'images Docker. Docker Hub est le registry public par défaut. On peut aussi utiliser un registry privé (AWS ECR, GitLab Registry...).

Layer

Chaque instruction d'un Dockerfile crée un layer (couche). Les layers sont mis en cache et partagés entre images — ce qui rend les builds et les téléchargements très efficaces.

Dockerfile — instructions clés

Exemple de Dockerfile

# Image de base
FROM python:3.11-slim

# Métadonnées
LABEL maintainer="bigdata-prep"

# Répertoire de travail dans le conteneur
WORKDIR /app

# Copier les fichiers de dépendances
COPY requirements.txt .

# Installer les dépendances
RUN pip install --no-cache-dir -r requirements.txt

# Copier le code source
COPY . .

# Port exposé par le conteneur
EXPOSE 8080

# Commande exécutée au démarrage
CMD ["python", "app.py"]

Instructions Dockerfile essentielles

InstructionRôle
FROMImage de base (obligatoire, première instruction)
WORKDIRRépertoire de travail pour les commandes suivantes
COPYCopie des fichiers de l'hôte vers le conteneur
RUNExécute une commande lors du build (crée un layer)
ENVDéfinit une variable d'environnement
EXPOSEDocumente le port (ne l'ouvre pas réellement)
CMDCommande par défaut au démarrage du conteneur (remplaçable)
ENTRYPOINTCommande fixe au démarrage (non remplaçable)
VOLUMEDéclare un point de montage pour la persistance

Commandes Docker essentielles

Gestion des images

# Builder une image depuis le Dockerfile du dossier courant
docker build -t mon-app:v1 .

# Lister les images locales
docker images

# Supprimer une image
docker rmi mon-app:v1

# Télécharger une image depuis Docker Hub
docker pull postgres:15

Gestion des conteneurs

# Lancer un conteneur (détaché, port mappé)
docker run -d -p 5432:5432 --name pg-local postgres:15

# Lister les conteneurs actifs
docker ps

# Lister tous les conteneurs (actifs + stoppés)
docker ps -a

# Arrêter / supprimer un conteneur
docker stop pg-local
docker rm pg-local

# Exécuter une commande dans un conteneur actif
docker exec -it pg-local psql -U postgres

# Afficher les logs d'un conteneur
docker logs pg-local

# Inspecter un conteneur (réseau, volumes, config)
docker inspect pg-local

Volumes et réseaux

# Créer un volume nommé
docker volume create pg-data

# Lancer un conteneur avec un volume
docker run -d -v pg-data:/var/lib/postgresql/data postgres:15

# Créer un réseau interne
docker network create mon-reseau

# Connecter un conteneur à un réseau
docker run -d --network mon-reseau --name worker1 postgres:15

Docker Compose

Définition

Outil permettant de décrire et lancer un ensemble de conteneurs dans un fichier YAML (docker-compose.yml). Indispensable pour orchestrer un cluster multi-conteneurs comme Citus (coordinateur + workers).

Exemple — Cluster Citus (coordinateur + 2 workers)

# docker-compose.yml
version: '3.8'

services:
  coordinator:
    image: citusdata/citus:12.1
    ports:
      - "5432:5432"
    environment:
      POSTGRES_PASSWORD: postgres
    volumes:
      - coordinator-data:/var/lib/postgresql/data
    networks:
      - citus-net

  worker1:
    image: citusdata/citus:12.1
    environment:
      POSTGRES_PASSWORD: postgres
    volumes:
      - worker1-data:/var/lib/postgresql/data
    networks:
      - citus-net

  worker2:
    image: citusdata/citus:12.1
    environment:
      POSTGRES_PASSWORD: postgres
    volumes:
      - worker2-data:/var/lib/postgresql/data
    networks:
      - citus-net

volumes:
  coordinator-data:
  worker1-data:
  worker2-data:

networks:
  citus-net:
    driver: bridge

Commandes Docker Compose essentielles

# Démarrer tous les services (en arrière-plan)
docker compose up -d

# Afficher l'état des services
docker compose ps

# Voir les logs de tous les services
docker compose logs -f

# Exécuter une commande dans un service
docker compose exec coordinator psql -U postgres

# Arrêter les services (sans supprimer)
docker compose stop

# Arrêter et supprimer les conteneurs + réseaux
docker compose down

# Arrêter + supprimer volumes (données perdues !)
docker compose down -v

DNS interne Docker

Dans un réseau Docker Compose, chaque service est accessible par son nom de service. Dans le cluster Citus, le coordinateur accède aux workers via worker1 et worker2 — sans adresse IP.

Docker Swarm

Mode d'orchestration natif Docker pour plusieurs machines physiques. Un Swarm manager orchestre des Docker daemons sur plusieurs nœuds. Plus simple que Kubernetes mais moins puissant.

Docker dans le contexte du cours

Lien avec Citus

Le TP Citus utilise Docker Compose pour simuler un cluster distribué sur une seule machine : un conteneur coordinateur et N conteneurs workers, tous sur le même réseau interne Docker (citus-net). Le coordinateur enregistre les workers via citus_add_node('worker1', 5432) — Docker résout le nom worker1 en adresse IP interne automatiquement.

Avantages de Docker pour les clusters de test

  • Reproducible : même environnement sur toutes les machines
  • Léger : pas besoin de 3 machines physiques ou VMs
  • Propre : docker compose down -v supprime tout sans laisser de traces
  • Rapide à démarrer : secondes, pas minutes
