Module 01 · Hadoop

Hadoop, HDFS & MapReduce

Le socle historique du Big Data : stockage distribué tolérant aux pannes (HDFS), gestion des ressources (YARN), modèle de traitement parallèle (MapReduce), et l'écosystème d'outils qui gravite autour.

Big Data — rappel

Définition

Volumes de données si massifs, variés et rapides qu'ils dépassent les capacités des outils de gestion classiques (SGBD relationnels mono-serveur).

VolumeVélocité VariétéVéracitéValeur

Scale-up (vertical)

Renforcer une seule machine (plus de CPU/RAM/disque). Limite physique et coût exponentiel.

Scale-out (horizontal)

Ajouter des machines standard (commodity hardware) au cluster. Approche d'Hadoop : économique et quasi illimitée.

Apache Hadoop — vue d'ensemble

Définition

Framework open source Java pour le stockage et le traitement distribué de très grands ensembles de données sur des clusters de serveurs standard. Inspiré des papiers Google GFS (2003) et MapReduce (2004), créé par Doug Cutting (issu du projet Nutch).

3 avantages majeurs

  • Tolérance aux pannes — réplication des données
  • Évolutivité horizontale — scale-out
  • Rentabilité — commodity hardware

3 piliers du cours

HDFS (stockage) · YARN (gestion des ressources) · MapReduce (traitement) — chacun a une architecture maître/esclave.

Architecture Master-Slave du cluster

Master Node

Orchestre le cluster : gère les métadonnées, coordonne les esclaves, ne stocke jamais les données réelles. Composants : NameNode (HDFS), ResourceManager (YARN), Secondary NameNode (checkpoints).

⚠️ Piège classique

Le Secondary NameNode n'est PAS un NameNode de secours. Il effectue des checkpoints périodiques des métadonnées (fusion du edit log et du fsimage) pour éviter la corruption — il ne peut pas prendre le relais en cas de panne.

Slave / Worker Nodes

Stockent les données réelles et exécutent les calculs. Composants : DataNode (HDFS), NodeManager (YARN), TaskTracker (déprécié, MapReduce v1).

Heartbeat

Chaque DataNode envoie régulièrement un signal de vie au NameNode. Sans heartbeat pendant 10 minutes, le nœud est déclaré défaillant et ses blocs sont re-répliqués ailleurs.

Haute disponibilité (HA)

Le problème : SPOF

Le NameNode est un point unique de défaillance. S'il tombe, tout le cluster devient inaccessible.

La solution : Active / Standby

Active NameNode traite les requêtes. Standby NameNode est synchronisé en temps réel et prêt à prendre le relais (failover automatique ou manuel).

Les briques de la HA

ComposantRôle
JournalNodesStockage partagé (cluster de 3 ou 5) où les deux NameNodes écrivent — assure la cohérence des métadonnées
ZooKeeperCoordonne le basculement automatique, élit le NameNode actif, évite le split-brain (deux actifs simultanés)

HDFS — Hadoop Distributed File System

Principe

Inspiré du Google File System (GFS, 2003). Un fichier est divisé en blocs de 128 Mo (par défaut), chaque bloc est répliqué (3 réplicats par défaut) et distribué sur plusieurs DataNodes.

NameNode

Gère l'arborescence, la localisation des blocs, les permissions. Ne contient que des métadonnées.

DataNode

Stocke les blocs réels, répond aux lectures/écritures, crée/supprime/réplique les blocs sur ordre du NameNode.

Workflow d'accès

1. Le client contacte le NameNode pour connaître l'arborescence et la localisation des blocs.
2. Le client accède directement aux DataNodes pour lire/écrire les données (le NameNode n'est jamais sur le chemin des données).

HDFS — commandes CLI

Structure générale : hdfs dfs -<commande> [arguments] — très proche d'UNIX.

# Navigation et gestion des dossiers
hdfs dfs -ls /user/data
hdfs dfs -mkdir /mon_projet
hdfs dfs -rm /data/old_file.txt
hdfs dfs -rm -r /temp_data

# Transfert local ↔ HDFS
hdfs dfs -put /home/user/dataset.csv /data/
hdfs dfs -get /data/resultats.csv ./local_folder/

# Consultation
hdfs dfs -cat /data/config.xml
hdfs dfs -tail /data/output.log
hdfs dfs -du -h /user/data

# Administration du cluster
hdfs dfsadmin -report   # état des DataNodes, espace libre
hdfs fsck /              # vérifie les blocs corrompus

YARN — Yet Another Resource Negotiator

Pourquoi YARN ?

Introduit dans Hadoop 2.0. Dans Hadoop 1.x, le JobTracker gérait à la fois les ressources ET les jobs → goulot d'étranglement et SPOF. YARN sépare ces deux responsabilités et devient un véritable système d'exploitation distribué, capable de faire tourner MapReduce, Spark, Flink...

ResourceManager (RM)

Le coordinateur global : gère les ressources du cluster (CPU/RAM), planifie et alloue les conteneurs.

NodeManager (NM)

Un par nœud esclave. Lance et surveille les conteneurs, rapporte l'utilisation des ressources au RM.

ApplicationMaster (AM)

Un par application (job). Négocie les ressources avec le RM et coordonne l'exécution des tâches.

Workflow

Client → soumet à RM → RM alloue un conteneur pour l'AM → l'AM négocie ses ressources → RM alloue des conteneurs sur les NM → l'AM lance les tâches → les NM surveillent et rapportent au RM → l'AM rapporte au client → fin, libération des ressources.

MapReduce

Définition

Modèle de programmation pour le traitement distribué, développé par Google (2004), inspiré de la programmation fonctionnelle. Principe : diviser (Map) → traiter en parallèle → combiner (Reduce).

Phase Map

Traite les données en entrée, produit des paires clé-valeur intermédiaires, exécutée en parallèle.

Shuffle & Sort

Étape intermédiaire : regroupement des valeurs par clé, tri, distribution aux reducers.

Phase Reduce

Agrège les valeurs par clé, produit le résultat final. Une tâche Reduce par clé.

WordCount — exemple

def mapper(line):
    for word in line.split():
        emit(word, 1)
# "Hello World Hello" → (Hello,1) (World,1) (Hello,1)

def reducer(key, values):
    emit(key, sum(values))
# Hello -> [1,1]  =>  (Hello, 2)

Avantages

  • Parallélisation automatique
  • Tolérance aux pannes
  • Scalabilité horizontale

Limites

  • Latence élevée (batch, écritures disque à chaque étape)
  • Pas adapté au temps réel
  • Remplacé en pratique par Spark (en mémoire, 10-100× plus rapide)

HBase

Base de données NoSQL distribuée, orientée colonnes, construite sur HDFS. Modèle clé-valeur avec familles de colonnes, accès aléatoire rapide en lecture/écriture, scalabilité linéaire. Inspirée de Google BigTable.

Hive

Data warehouse fournissant une interface SQL (HiveQL) sur des données stockées dans HDFS. Convertit les requêtes en jobs MapReduce/Spark. Métadonnées dans un metastore (souvent MySQL). Schéma à la lecture (schema-on-read). Avantages : facilité pour utilisateurs SQL, intégration BI (Tableau, Power BI), formats variés (Parquet, ORC, JSON).

Tendances actuelles

Hadoop aujourd'hui

Toujours utilisé en entreprise (Yahoo, Meta, LinkedIn), base de nombreux data lakes, complémentaire au Cloud (AWS EMR, Azure HDInsight, GCP Dataproc).

Évolutions

Spark remplace progressivement MapReduce. Lakehouse = fusion data lake + data warehouse (Delta Lake, Iceberg). Streaming temps réel avec Flink, Kafka Streams.
