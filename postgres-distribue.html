PostgreSQL distribué

Trois approches progressives pour distribuer une base relationnelle : partitionnement local, distribution logique via FDW, et distribution réelle avec Citus.

Vue d'ensemble — les 3 approches

ApprocheOù sont les donnéesQui exécuteLimite principale
Partitionnement localUn seul serveur, sous-tables physiquesPostgreSQL localPas de distribution réelle — toujours mono-serveur
FDW (Foreign Data Wrapper)Serveurs distants (réseau)Serveur central + serveurs distantsRequêtes toujours pilotées par le central — si un distant tombe, ses données sont inaccessibles

Partie 1 — Partitionnement local

Principe

PostgreSQL divise une table en sous-tables physiques appelées partitions. Les données restent sur un seul serveur. Lors d'une requête filtrée sur la clé de partition, le planificateur n'interroge que la ou les partitions concernées : c'est le partition pruning.

Types de partitionnement

  • PARTITION BY LIST : valeurs explicites (ex: pays)
  • PARTITION BY RANGE : plages de valeurs (ex: dates)
  • PARTITION BY HASH : répartition par hachage

Partition DEFAULT

Capture toutes les lignes dont la valeur ne correspond à aucune partition explicite. Obligatoire pour ne pas perdre de données lors des insertions.

Stratégie de migration (table existante → partitionnée)

Une table standard ne peut pas être convertie directement. Procédure :

-- 1. Renommer l'ancienne table
ALTER TABLE commandes RENAME TO commandes_old;

-- 2. Créer la nouvelle table partitionnée
CREATE TABLE commandes (
  commande_id INT,
  client_id   INT,
  statut      VARCHAR(20),
  pays_client VARCHAR(50)
) PARTITION BY LIST (pays_client);

-- 3. Créer les partitions
CREATE TABLE commande_togo    PARTITION OF commandes FOR VALUES IN ('Togo');
CREATE TABLE commande_mali    PARTITION OF commandes FOR VALUES IN ('Mali');
CREATE TABLE commande_default PARTITION OF commandes DEFAULT;

-- 4. Migrer les données
INSERT INTO commandes (commande_id, client_id, statut, pays_client)
SELECT c.commande_id, c.client_id, c.statut, cl.pays
FROM commandes_old c
JOIN clients cl ON cl.client_id = c.client_id;

-- 5. Vérifier avec EXPLAIN ANALYZE
EXPLAIN ANALYZE SELECT * FROM commandes WHERE pays_client = 'Togo';

Inspecter la hiérarchie des partitions

-- Lister les tables (r=ordinaire, p=partitionnée, f=étrangère)
SELECT relname, relkind FROM pg_class
WHERE relkind IN ('r', 'f', 'p') AND relnamespace = 2200;

-- Hiérarchie parent-enfant
SELECT inhparent::regclass AS parent, inhrelid::regclass AS enfant
FROM pg_inherits;

Partie 2 — Foreign Data Wrapper (FDW)

Principe

FDW permet à une partition de résider physiquement sur un serveur distant, tout en restant interrogeable de façon transparente depuis le serveur central. Le serveur central reste le point d'entrée unique. Scénario du TP : données Togo → serveur Lomé (port 5433), données Mali → serveur Bamako (port 5434).

⚠️ Limite critique

Si le serveur distant tombe en panne, les données qui y résident deviennent totalement inaccessibles depuis le serveur central. Les données des autres partitions (locales ou autres serveurs distants) restent disponibles.

Configuration FDW — 4 étapes

-- Étape 1 : activer l'extension
CREATE EXTENSION IF NOT EXISTS postgres_fdw;

-- Étape 2 : déclarer le serveur distant
CREATE SERVER server_lome
  FOREIGN DATA WRAPPER postgres_fdw
  OPTIONS (host '127.0.0.1', dbname 'flowshop_lome', port '5433');

-- Étape 3 : mapping utilisateur (authentification)
CREATE USER MAPPING FOR CURRENT_USER
  SERVER server_lome
  OPTIONS (user 'tt', password 'admin');

-- Étape 4 : créer la foreign table et l'attacher
CREATE FOREIGN TABLE commande_togo (
  commande_id INT, client_id INT,
  statut VARCHAR(20), pays_client VARCHAR(50)
) SERVER server_lome OPTIONS (table_name 'commande_togo');

ALTER TABLE commandes
  ATTACH PARTITION commande_togo FOR VALUES IN ('Togo');

Migrer une partition vers un serveur distant

-- Exporter la partition locale et l'importer sur le serveur distant
pg_dump -p 5432 -d flowshop_db -t commande_togo | psql -p 5433 -d flowshop_lome

-- Détacher et supprimer la partition locale devenue inutile
ALTER TABLE commandes DETACH PARTITION commande_togo;
DROP TABLE commande_togo;

Pourquoi co-localiser les données ?

Dans le TP, details_togo est migré sur le même serveur que commande_togo (Lomé) pour éviter les jointures distribuées coûteuses : les jointures entre ces deux tables s'exécutent alors localement sur le serveur de Lomé.

Vérifier l'exécution distante

EXPLAIN (VERBOSE, COSTS OFF)
SELECT * FROM details_commande
WHERE pays_client = 'Togo' LIMIT 5;
-- Le plan doit montrer "Foreign Scan on server_lome"

Partie 3 — Citus

Qu'est-ce que Citus ?

Extension PostgreSQL qui transforme une instance en base de données distribuée. Contrairement à FDW où les requêtes sont exécutées sur le serveur central, Citus parallélise les requêtes sur les workers. Les workers stockent des fragments de données appelés shards.

Coordinateur

Reçoit toutes les requêtes SQL. Les analyse, les planifie, les distribue aux workers appropriés, et agrège les résultats. C'est l'unique point d'entrée pour les clients.

Workers

Stockent les fragments de données (shards). Exécutent les sous-requêtes que le coordinateur leur délègue. Communiquent entre eux via un réseau interne.

Shard

Fragment horizontal d'une table distribuée, stocké sur un worker. Le coordinateur sait quel worker possède quel shard grâce aux métadonnées de distribution.

Types de tables Citus

TypeDescriptionCas d'usage
Distributed tableFragmentée horizontalement sur les workers selon une colonne de distributionGrandes tables de faits (commandes, transactions)
Reference tableRépliquée intégralement sur chaque workerPetites tables de référence (pays, catégories) pour les jointures locales
Local tableStockée uniquement sur le coordinateurTables de configuration, métadonnées Citus

Mise en place d'une table distribuée

-- Sur le coordinateur : activer Citus
CREATE EXTENSION citus;

-- Enregistrer les workers
SELECT citus_add_node('worker1', 5432);
SELECT citus_add_node('worker2', 5432);

-- Créer la table normalement
CREATE TABLE commandes (
  commande_id BIGINT, client_id INT,
  date_commande TIMESTAMP, montant DECIMAL
);

-- La distribuer sur les workers (32 shards par défaut)
SELECT create_distributed_table('commandes', 'client_id');

-- Créer une reference table (répliquée partout)
SELECT create_reference_table('pays');

Colocation

Deux tables distribuées sur la même colonne sont colocalisées : leurs shards correspondants sont sur le même worker. Les jointures entre ces tables s'exécutent localement sur chaque worker, sans transfert réseau.

Docker dans le contexte Citus

Le TP utilise Docker Compose pour simuler le cluster Citus : un conteneur coordinateur et N conteneurs workers, sur un réseau interne Docker. Chaque conteneur se voit par son nom (ex: coordinator, worker1).
